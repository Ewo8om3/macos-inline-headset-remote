# Contributing

Thank you for improving macOS Inline Headset Remote. Small, reviewable changes with clear safety behavior are preferred.

By participating, you agree to follow the [Code of Conduct](CODE_OF_CONDUCT.md).

## Before opening a change

- Search existing issues and pull requests.
- For a device report, collect sanitized `detect` and EventViewer evidence.
- For a preset, confirm the target application's shortcut from its current official documentation or settings.
- Do not include complete personal Karabiner configurations, backup snapshots, transcripts, credentials, or account data.
- Do not add dependencies unless the benefit clearly outweighs installation and audit cost.

## Development requirements

- macOS for live integration testing;
- Zsh;
- `jq`;
- Karabiner-Elements 15.4 or newer for device testing.

Clone the repository and run the checks:

```zsh
tests/validate-rules.zsh
tests/run.zsh
```

Also run Zsh syntax checks on changed scripts:

```zsh
zsh -n bin/headset-remote tests/*.zsh
```

## Change principles

- Preserve unrelated profiles, rules, devices, and parameters.
- Keep install and uninstall idempotent.
- Make every configuration mutation previewable with `--dry-run`.
- Create a valid backup before replacing live configuration.
- Never silently accept a generic `0/0` collision.
- Keep `--allow-disconnected` separate from generic-match acknowledgment.
- Reject malformed input before writing.
- Prefer declarative device and preset data over application-specific branches in the CLI.
- Avoid shell-command outputs in bundled presets.

## Adding a device definition

Add one file under `devices/` using a stable lowercase kebab-case ID. Base every identifier on observed Karabiner-EventViewer or CLI output.

Document:

- product and manufacturer labels;
- connection transport and adapter, if relevant;
- Karabiner identifier fields;
- whether the IDs are stable across reconnects;
- the consumer events generated by each button;
- tested macOS and Karabiner versions;
- known collision risks.

A `0/0` definition must be marked generic and must never be described as uniquely identifying a brand or model.

## Adding a preset

See [Creating presets](docs/creating-presets.md) for the file format and validation checklist. Keep one preset in `presets/<id>/`, including its manifest and Karabiner rule. A preset must use a stable ID and reference an existing device definition.

Safe presets should:

- use standard Karabiner key codes;
- scope every manipulator with the declared `device_if` condition;
- declare the same condition consistently across all three buttons;
- avoid secrets and machine-specific paths;
- avoid `shell_command` and irreversible shortcuts;
- describe application prerequisites and shortcut assumptions;
- remain useful if copied for manual import.

## Tests

Add or update fixtures for behavior changes. At minimum, cover:

- successful installation;
- dry-run without filesystem mutation;
- repeated installation without duplicates;
- preservation of unrelated configuration;
- collision and disconnected-device refusals;
- explicit override behavior;
- targeted uninstall;
- valid backup and guarded restore;
- malformed input rejection;
- `doctor` success and failure states.

Tests must not read or write the developer's live `~/.config/karabiner` directory.

## Documentation

Update the README and relevant guide for changes to commands, flags, safety behavior, preset mappings, or compatibility. Put user-visible changes in [CHANGELOG.md](CHANGELOG.md).

## Pull requests

In the pull request, explain:

- the user problem;
- the chosen behavior and alternatives considered;
- safety implications;
- exact verification performed;
- hardware and software used for live testing, if any;
- anything not tested.

Keep generated snapshots and personal configurations out of commits. Maintainers may ask for a smaller change if a pull request combines unrelated work.
